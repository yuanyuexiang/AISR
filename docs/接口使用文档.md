# AISR 接口使用文档(面向应用层)

> 本文档面向**上层应用**(Agent / 脚本 / IDE 插件)的开发者,描述如何调用 AISR
> 提供的统一接口。设计与定位见 [../技术方案.md](../技术方案.md)。
>
> AISR 只负责"会话运行时";Agent / Workflow / RAG 等逻辑由你的应用实现。

---

## 0. 三种接入方式

| 方式 | 适用场景 | 流式 | 跨语言 |
|------|---------|------|--------|
| **HTTP API**(本地 daemon) | Agent、非 Go 应用、需要多轮 + 流式 | ✅ NDJSON | ✅ |
| **Go SDK** | Go 应用 | ✅ `<-chan Event` | Go |
| **CLI / Python 客户端** | 脚本、简单一次性集成 | ✅ NDJSON | ✅ |

三种方式共享**同一套核心概念与事件模型**(见 §1、§2)。HTTP API 是规范契约,
SDK / CLI 是其上的封装。

承载方式:运行 `aisr serve` 启动本地 daemon,默认监听 Unix socket
`~/.aisr/aisr.sock`。SDK 与 Python 客户端默认连接该 socket。

---

## 1. 核心概念

| 概念 | 说明 |
|------|------|
| **Provider** | 一个被纳管的 AI CLI(`claude` / `cursor` / `gemini`)。 |
| **Session** | 一次持久化的会话,身份 = `session-id + workspace + 元数据`。进程是可丢弃的执行载体,默认按需拉起并 `--resume` 恢复上下文。 |
| **Workspace** | 会话绑定的工作目录。 |
| **Event** | 从底层 CLI 归一化出的结构化事件(见 §2),所有接口统一返回。 |
| **Capabilities** | 某 Provider 支持的能力,应用据此降级处理。 |

> 重要:Session 的生命周期独立于进程。即使没有常驻进程,只要 session-id 存在,
> 下一次调用就能恢复上下文。应用只需持有 session-id。

---

## 2. 统一事件模型(关键)

所有"发送 prompt"的接口都返回**一串结构化事件**,每个事件是一行 JSON
(NDJSON)。事件结构:

```jsonc
{
  "kind": "text",          // 事件类型,见下表
  "text": "正在分析项目…",  // 文本类事件的增量内容(其它类型可能为空)
  "raw":  { /* ... */ }    // provider 原始负载,保留以防信息丢失
}
```

| kind | 含义 | 典型字段 |
|------|------|---------|
| `text` | 模型文本输出(增量) | `text` |
| `thinking` | 模型推理过程(extended thinking) | `text`(推理文本)/ `raw`(含 signature) |
| `tool_use` | 发起一次工具调用 | `raw`(工具名 / 入参) |
| `tool_result` | 工具调用结果 | `raw` |
| `usage` | token / 费用用量 | `raw.input_tokens` / `raw.output_tokens` |
| `error` | 出错(本轮可能中止) | `text`(消息) / `raw.code`(错误码) |
| `done` | 本轮回答结束 | `raw.session_id` / `raw.stop_reason` |

一轮典型事件流:

```jsonl
{"kind":"text","text":"我先看一下目录结构。"}
{"kind":"tool_use","raw":{"name":"list_dir","input":{"path":"."}}}
{"kind":"tool_result","raw":{"name":"list_dir","output":"main.go\ngo.mod"}}
{"kind":"text","text":"这是一个 Go 项目,入口在 main.go。"}
{"kind":"usage","raw":{"input_tokens":1820,"output_tokens":143}}
{"kind":"done","raw":{"session_id":"dev-001","stop_reason":"end_turn"}}
```

> 不是所有 provider 都能产出全部事件类型;以该 provider 的 `Capabilities` 为准。
> PTY 退化模式下可能只有 `text` / `error` / `done`。

## 3. HTTP API(本地 daemon)

### 3.1 启动与连接

```bash
aisr serve                       # 监听 ~/.aisr/aisr.sock(Unix socket)
# 或可选 TCP(需 token,见 §3.6):
# aisr serve --listen 127.0.0.1:7878
```

Unix socket 的访问控制依赖文件权限(默认 0600,仅当前用户)。

> 构建二进制、前提(CLI 登录)、以及 **macOS / Windows 的部署与开机自启**见
> [§10 部署到本机](#10-部署到本机macos--windows)。

所有端点以 `/v1` 为前缀。请求体与响应体均为 JSON;发送 prompt 的端点响应为
NDJSON 流(`Content-Type: application/x-ndjson`)。

### 3.2 端点一览

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/v1/providers` | 列出 provider 及其 capabilities |
| POST | `/v1/sessions` | 创建 session |
| GET | `/v1/sessions` | 列出 session |
| GET | `/v1/sessions/{id}` | session 详情 |
| DELETE | `/v1/sessions/{id}` | 销毁 session(含持久化 context) |
| POST | `/v1/sessions/{id}/messages` | 发送一轮 prompt,**流式返回事件** |
| POST | `/v1/sessions/{id}/cancel` | 取消当前进行中的一轮 |

### 3.3 列出 Provider

```http
GET /v1/providers
```

```json
{
  "providers": [
    {
      "name": "claude",
      "capabilities": {
        "structured_output": true,
        "streaming": true,
        "resume": true,
        "tool_use": true,
        "mcp": true
      }
    }
  ]
}
```

### 3.4 创建 / 查看 / 删除 Session

```http
POST /v1/sessions
Content-Type: application/json

{ "provider": "claude", "workspace": "./demo" }
```

```json
{
  "id": "dev-001",
  "provider": "claude",
  "workspace": "/abs/path/demo",
  "status": "idle",
  "created_at": "2026-06-27T10:00:00Z"
}
```

```http
GET /v1/sessions
```

```json
{
  "sessions": [
    { "id": "dev-001", "provider": "claude", "status": "idle" },
    { "id": "go-001",  "provider": "cursor", "status": "active" }
  ]
}
```

```http
DELETE /v1/sessions/dev-001        ->  204 No Content
```

### 3.5 发送 Prompt(流式)

```http
POST /v1/sessions/dev-001/messages
Content-Type: application/json

{ "prompt": "优化这个 Go 项目" }
```

响应为 NDJSON 流,每行一个 §2 的事件,直到 `done`(或 `error`)。

curl 示例(经 Unix socket):

```bash
curl --unix-socket ~/.aisr/aisr.sock \
  -X POST http://localhost/v1/sessions/dev-001/messages \
  -H 'Content-Type: application/json' \
  -d '{"prompt":"优化这个 Go 项目"}' --no-buffer
```

```jsonl
{"kind":"text","text":"我先看看项目结构…"}
{"kind":"text","text":"建议把 handler 拆分为独立包。"}
{"kind":"done","raw":{"session_id":"dev-001","stop_reason":"end_turn"}}
```

#### Agent 模式(可选 `agent` 块,Claude + Cursor)

`messages` 请求体可带一个 `agent` 块,把自主工具型 agent 的旋钮透传给 CLI。
让上层能驱动一个会自己调工具、连外部 MCP、在指定目录里读写的 agent(而非纯
"prompt 进 / text 出")。**工具白名单 `allowed_tools` 就是权限闸,无需
`bypassPermissions`。** 字段都可选:

```http
POST /v1/sessions/dev-001/messages
Content-Type: application/json

{
  "prompt": "分析这个 deal 并在需要时问我",
  "agent": {
    "append_system_prompt": "你是 ABF 分析师…",
    "allowed_tools": ["Read", "Bash", "Skill", "mcp__auric__ask_user"],
    "disallowed_tools": [],
    "mcp_config": { "mcpServers": { "auric": { "type": "http", "url": "http://127.0.0.1:8000/mcp" } } },
    "add_dirs": ["/abs/deal", "/abs/ip-root"],
    "max_turns": 20,
    "permission_mode": null
  }
}
```

| 字段 | 含义 |
|------|------|
| `append_system_prompt` | 追加到默认 system prompt 之后 |
| `allowed_tools` / `disallowed_tools` | 工具白/黑名单(`Read` / `Bash` / `mcp__<server>__<tool>` / …) |
| `mcp_config` | 内联 MCP server JSON(CLI 的 `--mcp-config` 接受 JSON 字符串) |
| `add_dirs` | agent 工具可访问的额外目录(`--add-dir`) |
| `max_turns` | 内部工具循环上限(0 = CLI 默认) |
| `permission_mode` | 覆盖权限模式;一般留空,靠 `allowed_tools` 即可 |

> **Claude**:字段直接映射到 CLI flag(`--append-system-prompt` /
> `--allowedTools` / `--mcp-config` / `--add-dir` / `--max-turns` / …)。
>
> **Cursor**:也支持,但机制不同——`mcp_config` 会被写成 workspace 里的
> `.cursor/mcp.json`(+`--approve-mcps` / `--trust`),`system_prompt` /
> `append_system_prompt` 拼进 prompt;只有单个 `--workspace`(`add_dirs` 退化)、
> 无细粒度 `allowed_tools`(靠 `--force`),`model` 建议留空走 Auto(免费版只能
> Auto)。以各 provider 的 `capabilities.mcp` 为准。
>
> `agent` 块不影响普通(无 `agent`)调用。

中断进行中的一轮:`POST /v1/sessions/{id}/cancel` → `204`(无进行中的轮次时
返回 `409 NO_ACTIVE_TURN`)。

### 3.6 鉴权

* **Unix socket(默认)**:靠文件系统权限,无需 token。
* **TCP 模式**:必须带 `Authorization: Bearer <token>`;token 由 `aisr serve --token`
  或环境变量 `AISR_TOKEN` 指定(缺失则拒绝以 TCP 启动)。客户端可用 `AISR_BASE_URL` +
  `AISR_TOKEN` 环境变量配置。容器内调用见 [../docker/README.md](../docker/README.md)。

## 4. Go SDK

```go
import "github.com/<org>/aisr/pkg/sdk"

client := sdk.NewClient()                  // 默认连 ~/.aisr/aisr.sock
defer client.Close()

// 创建(或复用)会话
sess, err := client.CreateSession(ctx, sdk.SessionOpts{
    Provider:  "claude",
    Workspace: "./demo",
})

// 发送一轮,消费结构化事件流
events, err := client.Send(ctx, sess.ID, "优化这个 Go 项目")
for ev := range events {
    switch ev.Kind {
    case sdk.EventText:
        fmt.Print(ev.Text)
    case sdk.EventToolUse:
        // 处理工具调用事件
    case sdk.EventError:
        return ev.Err()
    case sdk.EventDone:
        // 本轮结束
    }
}
```

会话可跨多轮复用(自动 resume);持有 `sess.ID` 即可在进程重启后继续。

## 5. Python 客户端

```python
from aisr import Client

client = Client()                          # 默认连 ~/.aisr/aisr.sock

session = client.create_session(provider="claude", workspace="./demo")

for event in client.send(session.id, "优化这个 Go 项目"):
    if event.kind == "text":
        print(event.text, end="", flush=True)
    elif event.kind == "tool_use":
        ...
    elif event.kind == "error":
        raise RuntimeError(event.text)
    elif event.kind == "done":
        break
```

Python 客户端是薄封装:仅通过本地 socket 调用 HTTP API,不重复实现核心逻辑。

## 6. CLI(脚本集成)

适合不想起 daemon 的简单脚本。`--json` 让 `ask` 以 NDJSON 输出事件,便于解析:

```bash
SID=$(aisr session create --provider claude --workspace ./demo --quiet)  # 仅打印 id

aisr ask --session "$SID" --json "优化这个 Go 项目" | while IFS= read -r line; do
    kind=$(echo "$line" | jq -r .kind)
    [ "$kind" = "text" ] && echo "$line" | jq -rj .text
done

aisr session remove "$SID"
```

退出码:`0` 成功;非 `0` 表示出错(详见 §7,stderr 输出错误 JSON)。

## 7. 错误模型

错误以两种形式出现:

* **请求级错误**(如 session 不存在):HTTP 4xx/5xx + JSON 体;CLI 非零退出码。
* **流内错误**(如 provider 中途崩溃):事件流里的 `error` 事件。

统一错误结构:

```json
{ "error": { "code": "SESSION_NOT_FOUND", "message": "no session: dev-999" } }
```

| code | 含义 |
|------|------|
| `SESSION_NOT_FOUND` | session-id 不存在 |
| `PROVIDER_NOT_FOUND` | 未知 provider |
| `PROVIDER_UNAVAILABLE` | provider CLI 未安装 / 未登录 |
| `PROVIDER_CRASHED` | 底层进程异常退出 |
| `CAPABILITY_UNSUPPORTED` | 该 provider 不支持所请求能力 |
| `WORKSPACE_INVALID` | workspace 路径无效 |
| `RUNTIME_BUSY` | 达到 `max_sessions` 上限 |

## 8. 典型用法:Agent 集成

一个 Agent 的最小调用序列:

```text
1. GET  /v1/providers                          # 选 provider、检查 capabilities
2. POST /v1/sessions {provider, workspace}     # 拿到 session-id,持久化保存
3. 循环:
     POST /v1/sessions/{id}/messages {prompt}  # 流式消费事件
       - text        -> 累积/展示回答
       - tool_use    -> 按需联动应用侧逻辑(可选)
       - done        -> 进入下一轮
   (上下文由 AISR 通过 resume 自动维持,Agent 无需自己拼历史)
4. DELETE /v1/sessions/{id}                     # 任务结束后销毁
```

要点:
* **Agent 不需要自己管理对话历史**——交给 session 的 resume 机制。
* **能力差异要降级处理**:发请求前查 `capabilities`;若 `tool_use=false`,不要
  依赖工具事件。
* **长任务用流式**:不要等整轮结束;边收 `text`/`tool_use` 边处理。
* **持久化 session-id**:它是恢复上下文的唯一句柄,应用重启后仍可用。

---

## 9. 版本与稳定性

* HTTP API 以 `/v1` 前缀做版本管理;破坏性变更会升级前缀。
* 事件模型(§2)是跨三种接入方式的稳定契约;新增 `kind` 时,应用应对未知
  `kind` 做忽略处理,以保持向前兼容。

## 10. 部署到本机(macOS / Windows)

AISR 是一个**本机 daemon**,复用你已登录的 `claude` / `cursor-agent` CLI(零 API
key)。部署 = 拿到 `aisr` 二进制 + CLI 已登录 + 让 daemon 常驻(可选开机自启)。

> **定位提醒**:AISR 是**本地个人 runtime**,不是对外网关。TCP 模式**务必带
> token**,且尽量绑到 `127.0.0.1` 或 docker 网关而非 `0.0.0.0`(见 §3.6 与
> [../docker/README.md](../docker/README.md))。订阅授权面向个人交互式使用,包成对外
> 多用户服务可能违反 ToS。

### 10.1 前提

- 目标 CLI 已装好并**登录**:`claude -p "hi"`、`cursor-agent status` 能正常回应
  (认证是宿主机的事,与调用方无关)。
- 从源码构建需 **Go 1.26+**;或直接用交叉编译好的产物(`dist/`)。

### 10.2 拿到二进制

| 场景 | 命令 |
|---|---|
| macOS / Linux 本机构建 | `go build -o bin/aisr ./cmd/aisr`(或 `make build`) |
| Windows 本机构建(PowerShell) | `go build -o bin\aisr.exe ./cmd/aisr` |
| 在 mac/Linux 交叉编译 Windows 产物 | `make windows` → `dist/aisr-windows-amd64.exe` / `-arm64.exe` |

### 10.3 macOS

**手动跑**(前台;`Ctrl-C` 优雅退出、清理 socket):

```bash
./bin/aisr serve                                  # 默认 unix socket(仅本机 Go/CLI/Python 客户端)
# 或 TCP(容器 / 跨语言 / 别的机器要连时):
AISR_TOKEN=$(openssl rand -hex 16) ./bin/aisr serve --listen 0.0.0.0:7878
```

**开机自启(launchd,用户级 LaunchAgent)** —— `~/Library/LaunchAgents/com.aisr.daemon.plist`:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN" "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0"><dict>
  <key>Label</key><string>com.aisr.daemon</string>
  <key>ProgramArguments</key>
  <array>
    <string>/Users/&lt;you&gt;/path/to/bin/aisr</string>
    <string>serve</string>
    <string>--listen</string><string>0.0.0.0:7878</string>
  </array>
  <key>EnvironmentVariables</key><dict>
    <key>AISR_TOKEN</key><string>换成强随机值</string>
    <!-- 关键:daemon 要能在 PATH 上找到 claude / cursor-agent。launchd 的 PATH
         很精简,把 CLI 所在目录写全(绝对路径,~ 不会展开)。 -->
    <key>PATH</key><string>/Users/&lt;you&gt;/.local/bin:/opt/homebrew/bin:/usr/local/bin:/usr/bin:/bin</string>
  </dict>
  <key>RunAtLoad</key><true/>
  <key>KeepAlive</key><true/>
  <key>StandardOutPath</key><string>/tmp/aisr.log</string>
  <key>StandardErrorPath</key><string>/tmp/aisr.err</string>
</dict></plist>
```

```bash
launchctl load   ~/Library/LaunchAgents/com.aisr.daemon.plist   # 立即启动 + 开机自启
launchctl unload ~/Library/LaunchAgents/com.aisr.daemon.plist   # 停止 + 取消自启
tail -f /tmp/aisr.err                                           # 看日志
```

> **常见坑**:daemon 起来了但调用报 `PROVIDER_UNAVAILABLE` —— 多半是 launchd 的
> `PATH` 里没有 `claude` / `cursor-agent` 的目录。`which claude` 查到目录填进上面的
> `PATH`(或用 `AISR_CLAUDE_BIN` / `AISR_CURSOR_BIN` 写 CLI 全路径)。

### 10.4 Windows(原生 PowerShell)

原生 Windows **一律用 TCP**(Python 的 `AF_UNIX` 通常不可用);provider 名可能是
`.cmd`。完整注意事项见 [windows.md](windows.md),精简步骤:

```powershell
# 1) 构建
go build -o bin\aisr.exe ./cmd/aisr

# 2) 确认 CLI 登录;命令名不同就覆盖(本会话内有效)
claude -p "hi"; cursor-agent status
$env:AISR_CLAUDE_BIN = "claude.cmd"
$env:AISR_CURSOR_BIN = "cursor-agent.cmd"

# 3) 起 daemon(TCP + token)
$env:AISR_TOKEN = "换成强随机值"
.\bin\aisr.exe serve --listen 0.0.0.0:7878
```

**开机自启** —— 先写一个启动脚本(避免 schtasks 里的嵌套引号地狱),`start-aisr.ps1`:

```powershell
$env:AISR_TOKEN = "换成强随机值"
& "C:\path\to\bin\aisr.exe" serve --listen 0.0.0.0:7878
```

登录时后台拉起(任务计划程序):

```powershell
schtasks /Create /TN "AISR Daemon" /SC ONLOGON /RL LIMITED `
  /TR "powershell -WindowStyle Hidden -File C:\path\to\start-aisr.ps1"
```

或用 **NSSM** 做成真正的 Windows 服务(开机即启、崩溃自动拉起):

```powershell
nssm install AISR C:\path\to\bin\aisr.exe serve --listen 0.0.0.0:7878
nssm set AISR AppEnvironmentExtra AISR_TOKEN=换成强随机值
nssm start AISR
```

> `taskkill` 不触发优雅退出(残留 socket 下次启动时会被清理);`Ctrl+C` 才优雅。
> **WSL2** 里则完全按 Linux / unix-socket 用法,不受本节限制。

### 10.5 验证部署

```bash
# macOS unix socket:
curl --unix-socket ~/.aisr/aisr.sock http://localhost/v1/providers
# TCP(mac / Windows 通用):
curl -H "Authorization: Bearer <token>" http://127.0.0.1:7878/v1/providers
```

返回 `providers` 列表、且目标 provider 的 `capabilities` 正常,即部署成功。
