# AISR 接口使用文档(面向应用层)

> 本文档面向**上层应用**(Agent / 脚本 / IDE 插件)的开发者,描述如何调用 AISR
> 提供的统一接口。设计与定位见 [../技术方案.md](../技术方案.md)。
>
> AISR 只负责"会话运行时";Agent / Workflow / RAG 等逻辑由你的应用实现。

---

## 0. 三种接入方式

| 方式 | 适用场景 | 流式 | 跨语言 |
|------|---------|------|--------|
| **HTTP API**(本地 daemon) | Agent、非 Go 应用、需要多轮 + 流式 | ✅ NDJSON | ✅ |
| **Go SDK** | Go 应用 | ✅ `<-chan Event` | Go |
| **CLI / Python 客户端** | 脚本、简单一次性集成 | ✅ NDJSON | ✅ |

三种方式共享**同一套核心概念与事件模型**(见 §1、§2)。HTTP API 是规范契约,
SDK / CLI 是其上的封装。

承载方式:运行 `aisr serve` 启动本地 daemon,默认监听 Unix socket
`~/.aisr/aisr.sock`。SDK 与 Python 客户端默认连接该 socket。

---

## 1. 核心概念

| 概念 | 说明 |
|------|------|
| **Provider** | 一个被纳管的 AI CLI(`claude` / `cursor` / `gemini`)。 |
| **Session** | 一次持久化的会话,身份 = `session-id + workspace + 元数据`。进程是可丢弃的执行载体,默认按需拉起并 `--resume` 恢复上下文。 |
| **Workspace** | 会话绑定的工作目录。 |
| **Event** | 从底层 CLI 归一化出的结构化事件(见 §2),所有接口统一返回。 |
| **Capabilities** | 某 Provider 支持的能力,应用据此降级处理。 |

> 重要:Session 的生命周期独立于进程。即使没有常驻进程,只要 session-id 存在,
> 下一次调用就能恢复上下文。应用只需持有 session-id。

---

## 2. 统一事件模型(关键)

所有"发送 prompt"的接口都返回**一串结构化事件**,每个事件是一行 JSON
(NDJSON)。事件结构:

```jsonc
{
  "kind": "text",          // 事件类型,见下表
  "text": "正在分析项目…",  // 文本类事件的增量内容(其它类型可能为空)
  "raw":  { /* ... */ }    // provider 原始负载,保留以防信息丢失
}
```

| kind | 含义 | 典型字段 |
|------|------|---------|
| `text` | 模型文本输出(增量) | `text` |
| `tool_use` | 发起一次工具调用 | `raw`(工具名 / 入参) |
| `tool_result` | 工具调用结果 | `raw` |
| `usage` | token / 费用用量 | `raw.input_tokens` / `raw.output_tokens` |
| `error` | 出错(本轮可能中止) | `text`(消息) / `raw.code`(错误码) |
| `done` | 本轮回答结束 | `raw.session_id` / `raw.stop_reason` |

一轮典型事件流:

```jsonl
{"kind":"text","text":"我先看一下目录结构。"}
{"kind":"tool_use","raw":{"name":"list_dir","input":{"path":"."}}}
{"kind":"tool_result","raw":{"name":"list_dir","output":"main.go\ngo.mod"}}
{"kind":"text","text":"这是一个 Go 项目,入口在 main.go。"}
{"kind":"usage","raw":{"input_tokens":1820,"output_tokens":143}}
{"kind":"done","raw":{"session_id":"dev-001","stop_reason":"end_turn"}}
```

> 不是所有 provider 都能产出全部事件类型;以该 provider 的 `Capabilities` 为准。
> PTY 退化模式下可能只有 `text` / `error` / `done`。

## 3. HTTP API(本地 daemon)

### 3.1 启动与连接

```bash
aisr serve                       # 监听 ~/.aisr/aisr.sock(Unix socket)
# 或可选 TCP(需 token,见 §3.6):
# aisr serve --listen 127.0.0.1:7878
```

Unix socket 的访问控制依赖文件权限(默认 0600,仅当前用户)。

所有端点以 `/v1` 为前缀。请求体与响应体均为 JSON;发送 prompt 的端点响应为
NDJSON 流(`Content-Type: application/x-ndjson`)。

### 3.2 端点一览

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/v1/providers` | 列出 provider 及其 capabilities |
| POST | `/v1/sessions` | 创建 session |
| GET | `/v1/sessions` | 列出 session |
| GET | `/v1/sessions/{id}` | session 详情 |
| DELETE | `/v1/sessions/{id}` | 销毁 session(含持久化 context) |
| POST | `/v1/sessions/{id}/messages` | 发送一轮 prompt,**流式返回事件** |
| POST | `/v1/sessions/{id}/cancel` | 取消当前进行中的一轮 |

### 3.3 列出 Provider

```http
GET /v1/providers
```

```json
{
  "providers": [
    {
      "name": "claude",
      "capabilities": {
        "structured_output": true,
        "streaming": true,
        "resume": true,
        "tool_use": true,
        "mcp": false
      }
    }
  ]
}
```

### 3.4 创建 / 查看 / 删除 Session

```http
POST /v1/sessions
Content-Type: application/json

{ "provider": "claude", "workspace": "./demo" }
```

```json
{
  "id": "dev-001",
  "provider": "claude",
  "workspace": "/abs/path/demo",
  "status": "idle",
  "created_at": "2026-06-27T10:00:00Z"
}
```

```http
GET /v1/sessions
```

```json
{
  "sessions": [
    { "id": "dev-001", "provider": "claude", "status": "idle" },
    { "id": "go-001",  "provider": "cursor", "status": "active" }
  ]
}
```

```http
DELETE /v1/sessions/dev-001        ->  204 No Content
```

### 3.5 发送 Prompt(流式)

```http
POST /v1/sessions/dev-001/messages
Content-Type: application/json

{ "prompt": "优化这个 Go 项目" }
```

响应为 NDJSON 流,每行一个 §2 的事件,直到 `done`(或 `error`)。

curl 示例(经 Unix socket):

```bash
curl --unix-socket ~/.aisr/aisr.sock \
  -X POST http://localhost/v1/sessions/dev-001/messages \
  -H 'Content-Type: application/json' \
  -d '{"prompt":"优化这个 Go 项目"}' --no-buffer
```

```jsonl
{"kind":"text","text":"我先看看项目结构…"}
{"kind":"text","text":"建议把 handler 拆分为独立包。"}
{"kind":"done","raw":{"session_id":"dev-001","stop_reason":"end_turn"}}
```

### 3.6 鉴权

* **Unix socket(默认)**:靠文件系统权限,无需 token。
* **TCP 模式**:必须带 `Authorization: Bearer <token>`,token 见配置
  (`server.token`)。仅建议在受控环境开启 TCP。

## 4. Go SDK

```go
import "github.com/<org>/aisr/pkg/sdk"

client := sdk.NewClient()                  // 默认连 ~/.aisr/aisr.sock
defer client.Close()

// 创建(或复用)会话
sess, err := client.CreateSession(ctx, sdk.SessionOpts{
    Provider:  "claude",
    Workspace: "./demo",
})

// 发送一轮,消费结构化事件流
events, err := client.Send(ctx, sess.ID, "优化这个 Go 项目")
for ev := range events {
    switch ev.Kind {
    case sdk.EventText:
        fmt.Print(ev.Text)
    case sdk.EventToolUse:
        // 处理工具调用事件
    case sdk.EventError:
        return ev.Err()
    case sdk.EventDone:
        // 本轮结束
    }
}
```

会话可跨多轮复用(自动 resume);持有 `sess.ID` 即可在进程重启后继续。

## 5. Python 客户端

```python
from aisr import Client

client = Client()                          # 默认连 ~/.aisr/aisr.sock

session = client.create_session(provider="claude", workspace="./demo")

for event in client.send(session.id, "优化这个 Go 项目"):
    if event.kind == "text":
        print(event.text, end="", flush=True)
    elif event.kind == "tool_use":
        ...
    elif event.kind == "error":
        raise RuntimeError(event.text)
    elif event.kind == "done":
        break
```

Python 客户端是薄封装:仅通过本地 socket 调用 HTTP API,不重复实现核心逻辑。

## 6. CLI(脚本集成)

适合不想起 daemon 的简单脚本。`--json` 让 `ask` 以 NDJSON 输出事件,便于解析:

```bash
SID=$(aisr session create --provider claude --workspace ./demo --quiet)  # 仅打印 id

aisr ask --session "$SID" --json "优化这个 Go 项目" | while IFS= read -r line; do
    kind=$(echo "$line" | jq -r .kind)
    [ "$kind" = "text" ] && echo "$line" | jq -rj .text
done

aisr session remove "$SID"
```

退出码:`0` 成功;非 `0` 表示出错(详见 §7,stderr 输出错误 JSON)。

## 7. 错误模型

错误以两种形式出现:

* **请求级错误**(如 session 不存在):HTTP 4xx/5xx + JSON 体;CLI 非零退出码。
* **流内错误**(如 provider 中途崩溃):事件流里的 `error` 事件。

统一错误结构:

```json
{ "error": { "code": "SESSION_NOT_FOUND", "message": "no session: dev-999" } }
```

| code | 含义 |
|------|------|
| `SESSION_NOT_FOUND` | session-id 不存在 |
| `PROVIDER_NOT_FOUND` | 未知 provider |
| `PROVIDER_UNAVAILABLE` | provider CLI 未安装 / 未登录 |
| `PROVIDER_CRASHED` | 底层进程异常退出 |
| `CAPABILITY_UNSUPPORTED` | 该 provider 不支持所请求能力 |
| `WORKSPACE_INVALID` | workspace 路径无效 |
| `RUNTIME_BUSY` | 达到 `max_sessions` 上限 |

## 8. 典型用法:Agent 集成

一个 Agent 的最小调用序列:

```text
1. GET  /v1/providers                          # 选 provider、检查 capabilities
2. POST /v1/sessions {provider, workspace}     # 拿到 session-id,持久化保存
3. 循环:
     POST /v1/sessions/{id}/messages {prompt}  # 流式消费事件
       - text        -> 累积/展示回答
       - tool_use    -> 按需联动应用侧逻辑(可选)
       - done        -> 进入下一轮
   (上下文由 AISR 通过 resume 自动维持,Agent 无需自己拼历史)
4. DELETE /v1/sessions/{id}                     # 任务结束后销毁
```

要点:
* **Agent 不需要自己管理对话历史**——交给 session 的 resume 机制。
* **能力差异要降级处理**:发请求前查 `capabilities`;若 `tool_use=false`,不要
  依赖工具事件。
* **长任务用流式**:不要等整轮结束;边收 `text`/`tool_use` 边处理。
* **持久化 session-id**:它是恢复上下文的唯一句柄,应用重启后仍可用。

## 9. 版本与稳定性

* HTTP API 以 `/v1` 前缀做版本管理;破坏性变更会升级前缀。
* 事件模型(§2)是跨三种接入方式的稳定契约;新增 `kind` 时,应用应对未知
  `kind` 做忽略处理,以保持向前兼容。
