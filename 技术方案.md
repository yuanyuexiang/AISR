# AISR — AI Session Runtime

> **A unified session runtime for AI CLIs.**
> 一个运行在本机的 AI CLI 会话运行时:统一管理多个 AI CLI 的 Session 与上下文,
> 并对外提供统一的调用接口(CLI + SDK)。

## 核心动机

让本机上的**其他 agent 应用,无需自备 API key、也无需按 token 付费,就能用上大模型。**

做法:claude code / cursor / gemini 等 CLI 已经用各自的**订阅(subscription)登录**好了;
AISR 复用它们登录好的凭据,在 headless 模式下统一驱动它们,把模型能力以统一接口
(CLI / SDK / 本地 HTTP API)暴露给上层 agent。对上层而言:**零 API key、零 token 账单,
却能用上大模型**——这就是 AISR 存在的理由。

## 定位边界(重要)

* **本地个人 runtime**:服务对象是「你自己的 agent + 你自己的订阅 + 你自己的机器」。
* **不是对外/共享的 LLM 网关**:不要把它包装成对多人 / 对外提供服务的 API 网关——
  订阅授权一般面向个人交互式使用,对外提供服务可能违反服务条款(ToS)并导致封号。
* 由此自然推出它**不负责** Agent / Workflow / RAG / Web UI(这些都是上层应用)。

### 已知约束

1. **ToS / 账号风险**:仅限本机个人自用;对外服务、多人共用同一订阅会踩 ToS 红线。
2. **额度 / 限流**:订阅有用量上限,多 agent 共用比单人交互更易撞限流,不像付费 API
   那样能横向扩展。
3. **是 CLI 行为,而非裸模型端点**:输出带各 CLI 自身的系统提示 / 风格,可用 `--model`、
   `--system-prompt` 等调节,但与直连 API 的原始模型行为会有差异。

---

本项目**不是一个 AI Agent**,而是一个 **AI CLI Runtime**。它的职责只有一个:

> **统一管理 AI CLI 的 Session,并向其他程序提供统一调用接口。**

去掉 Docker、Web UI、React 之后,整个项目更轻量,也更容易开源。定位足够窄、
足够稳:它只做"会话运行时"这一层,把 Agent / Workflow / RAG 等都留给上层应用。

# 二、设计原则

这几条是整个方案能否落地的关键,优先级高于具体实现。

### 原则 1:优先"结构化集成",而非"解析终端" ⭐

目标 CLI 多是 TUI 程序,stdout 里混有 ANSI 转义、spinner、进度条、TUI 重绘,
直接解析极其脆弱,且会随 CLI 升级而碎掉;交互式程序还常检测 isatty,用普通
pipe 接它们会触发非交互降级或异常缓冲。

因此 AISR **优先对接各 CLI 的"非交互 / 无头 / 结构化输出"模式**(JSON 事件流),
把"PTY + 文本解析"仅作为没有结构化模式时的**退化方案(best-effort)**,并显式
标注其可靠性较低。

> 示例:Claude Code 提供 `claude -p "<prompt>" --output-format stream-json`,
> 直接输出结构化 JSON 事件流,天然就是我们想要的 Chunk,无需解析 TUI。

### 原则 2:接口要表达"能力差异",不做最低公分母

不同 provider 在工具调用、权限确认、文件编辑、MCP、用量统计、流式等能力上差异
很大。统一接口若只有"发文本 / 收文本",会把各 CLI 最有价值的能力全抹平。因此
Provider 必须**声明自身能力(Capabilities)**,并以**类型化事件**(text / tool_use /
tool_result / usage / error / done)对外输出,同时保留 provider 原始负载以防信息丢失。

### 原则 3:会话以"持久化 session-id"为中心,进程可丢弃

Session 的价值是**上下文保持**,而上下文可以持久化到磁盘。因此 AISR 把"会话"
的真身定义为 **持久化的 session-id + workspace + 元数据**,进程只是可丢弃的执行
载体。默认采用 **按需拉起 + `--resume` 恢复** 的轻量模型;"常驻热进程池"仅作为
减少冷启动延迟的**可选优化**(V2),而非默认架构。

### 原则 4:最小闭环优先

V1 先打通**一个 provider(Claude)** 的完整链路
(`create → ask → resume → list → remove`),验证抽象成立后再横向扩展到
Cursor / Gemini。先把一条链路做深,而不是一开始就铺三个 provider + 两套 SDK。

# 五、核心模块

## 1、Session Manager

负责会话的生命周期与索引:

```text
Create()    // 新建会话(分配 session-id + workspace + 元数据,落盘)
Acquire()   // 取得一个可用会话的执行句柄
Release()   // 释放执行句柄(默认即可让进程退出)
Destroy()   // 销毁会话(含持久化的 context)
Recover()   // 进程崩溃后,依据持久化的 session-id 恢复
```

维护**会话索引**(持久化在磁盘),例如:

```text
session-001   claude   ./workspace/ai     idle
session-002   cursor   ./workspace/go     active
```

> 注:这里是"会话索引",不等于"常驻进程池"。常驻进程是 V2 的可选优化。

## 2、Provider

Provider 把某个 AI CLI 归一化为统一的**结构化事件流**,并声明自身能力。

```go
// EventKind 是从底层 CLI 归一化出来的事件类型
type EventKind string

const (
    EventText       EventKind = "text"        // 模型文本输出(可增量)
    EventToolUse    EventKind = "tool_use"    // 发起工具调用
    EventToolResult EventKind = "tool_result" // 工具调用结果
    EventUsage      EventKind = "usage"       // token / 费用用量
    EventError      EventKind = "error"
    EventDone       EventKind = "done"        // 一轮回答结束
)

// Event 是对外暴露的统一事件;Raw 保留 provider 原始负载,避免信息丢失
type Event struct {
    Kind EventKind
    Text string
    Raw  json.RawMessage
}

// Capabilities 声明 provider 支持的能力,上层据此降级
type Capabilities struct {
    StructuredOutput bool // 是否有 JSON / 无头结构化输出(决定集成模式)
    Streaming        bool
    Resume           bool // 是否支持按 session-id 恢复上下文
    ToolUse          bool
    MCP              bool
}

type SessionOpts struct {
    SessionID string // 为空 = 新建;非空 = 尝试恢复
    Workspace string
}

type Provider interface {
    Name() string
    Capabilities() Capabilities

    // 开启或恢复一个会话,返回(可持久化的)session-id
    Start(ctx context.Context, opts SessionOpts) (sessionID string, err error)

    // 发送一轮 prompt,返回归一化后的结构化事件流
    Send(ctx context.Context, prompt string) (<-chan Event, error)

    // 结束本次执行(不一定销毁持久化的 context)
    Close() error
}
```

**集成模式分两类**,由 `Capabilities.StructuredOutput` 区分:

* **结构化模式(优先)**:对接 CLI 的无头 / JSON 输出,直接解析事件。
  * `ClaudeProvider`:`claude -p --output-format stream-json --verbose` + `--resume`
    (`--verbose` 为 stream-json 必需项,实测见 §十)
* **PTY 退化模式(best-effort)**:无结构化模式时,分配伪终端并尽力解析文本。
  * `CursorProvider` / `GeminiProvider`:**先做可行性验证**(见第十节)再定。

## 5、Stream Manager

统一把各 provider 的事件流归一化后,经 SDK 推送给应用:

```text
Provider(JSON 事件) → 归一化 Event → Stream Manager → SDK → Application
```

Claude / Cursor / Gemini 全部统一为同一套 `Event` 类型对外。

# 六、会话模型与生命周期

AISR 区分**会话状态**(持久,存盘)与**进程状态**(易失,可重建):

```text
会话状态:  Create → Idle → Active → Idle → Expired → Destroy
进程状态:  Cold ──(按需拉起)──> Warm ──(用完/超时)──> Cold
```

默认执行策略(轻量):

```text
Acquire()
   │  会话存在 → 按需拉起进程,用 --resume 接上下文
   ▼
Use(发送一轮 prompt,消费事件流)
   │
   ▼
Release()  → 进程退出,context 已持久化,会话回到 Idle
```

可选执行策略(V2 优化):`resident` 模式保持热进程,省去冷启动延迟,受
`max_resident` 限制。

# 八、对外 API 与 SDK

对外接口以**本地 daemon(`aisr serve`)+ Unix socket 上的 HTTP/JSON API** 为
规范契约;Go SDK / Python 客户端 / CLI 都是它的客户端。这与第二节整体架构
(中心 runtime + 多客户端)一致,也是 Python 等非 Go 应用能接入的前提。流式统一
采用 **NDJSON(逐行 JSON 事件)**,使三种接入方式吐出的事件格式字面一致。

* **HTTP API(规范契约)**:`/v1` 端点,事件流为 NDJSON
* **Go SDK(V1 核心)**:`client.Send(...)` 返回 `<-chan Event`
* **Python 客户端(V1 薄封装)**:仅调用本地 socket API,不重复实现核心
* **CLI**:`aisr ask --json ...` 输出 NDJSON,供脚本解析

> 完整的端点、事件模型与各语言示例见 [docs/接口使用文档.md](docs/接口使用文档.md)。
> V1 不同时维护两套"重" SDK:Go SDK 为核心,Python 仅做薄客户端。

# 十、集成可行性验证(Spike,先做)

整个方案的成败几乎只取决于一个问题:**目标 CLI 能否被程序稳定、结构化地驱动?**
因此在写正式代码前,先用最小 spike 逐一验证:

1. **Claude**:Go 起子进程跑 `claude -p "<prompt>" --output-format stream-json`,
   解析 JSON 事件流;验证 `--resume <session-id>` 能否跨调用接上上下文。
2. **Cursor**:确认是否存在等价的非交互 / 结构化输出模式。
3. **Gemini**:同上。

**产出**:一张能力矩阵,决定每个 provider 走"结构化"还是"PTY 退化"。

| Provider | 结构化输出 | Resume | 工具事件 | 流式 | 集成模式 |
|----------|-----------|--------|---------|------|---------|
| claude   | ✅ 已验证 | ✅ 已验证 | ✅(content 含 tool_use) | ✅ | **structured** |
| cursor   | 待验证 | 待验证 | 待验证 | 待验证 | 待定 |
| gemini   | 待验证 | 待验证 | 待验证 | 待验证 | 待定 |

**通过标准**:Claude 链路 `create → ask → resume → list` 跑通,即进入 V1。

## 实测结论(Claude,已通过 ✅)

实跑环境:claude_code_version 2.1.193 / macOS。

**正确命令**(踩坑):`--print` + `--output-format stream-json` **必须加 `--verbose`**,
否则报错 `requires --verbose`:

```bash
claude -p "<prompt>" --output-format stream-json --verbose
claude -p "<prompt>" --output-format stream-json --verbose --resume <session-id>
```

**输出为 NDJSON,事件类型与映射**:

| Claude 原始事件 | 关键字段 | 映射到统一 Event |
|----------------|---------|-----------------|
| `system` / `init` | `session_id`、`model`、`apiKeySource` | 内部捕获 session-id |
| `rate_limit_event` | `rate_limit_info.status`、`resetsAt`、`rateLimitType` | meta(对应额度/限流约束) |
| `assistant` | `message.content[]`(`text` / `tool_use`)、`usage` | `text` / `tool_use` |
| `result` | `result`、`stop_reason`、`session_id`、`usage`、`total_cost_usd` | `done` + `usage` |

**关键事实**:

1. **零 API key 成立**:init 事件 `"apiKeySource":"none"`,走订阅认证,印证核心动机。
2. **session-id 稳定**:`--resume` 前后 session-id 不变,可作稳定句柄;resume 会复用
   上下文缓存(第二轮 input token 显著下降)。
3. **resume 上下文延续已验证**:resume 后能准确复述上一轮的原始 prompt。
4. **每次调用有固定 harness 开销**:headless 每次都重新加载 Claude Code 系统提示
   (本次缓存创建 6–9k token,等价成本约 $0.08–0.10/次)。

**由实测推出的设计调整**:

* `ClaudeProvider` 内置命令须带 `--verbose`。
* 解析器以 `type` 字段分发(`system` / `rate_limit_event` / `assistant` / `result`);
  对未知 `type` 忽略,保持向前兼容。
* **常驻进程(resident)优先级上调**:on-demand 每次都付 harness 开销,常驻可摊薄;
  虽仍列 V2,但应尽早做。
* **开销优化(V1 待办)**:纯当 LLM 用时,考虑 `--model` 选更便宜模型、并裁剪加载的
  工具/skill,以压低每次调用的 token 开销。

> 关于 daemon 与"最小闭环优先"(原则 4):V1 仍把 daemon 纳入,是为了让对外接口

> 是 **provider 维度先只做 Claude**,而非砍掉对外接口。
